## PART 4: AI WORKFORCE SYSTEM — TECHNICAL GAPS

### 4.1 Knowledge Base is Mostly Unreadable by Agents

33 files in the knowledge base. The vast majority are PDFs, PPTX, and XLSX. Agents can read text files easily but struggle with binary formats. The catalog.py also misdetects language — it marks Arabic PDFs as "English" because it only checks text files for Arabic characters.

**Fix:**
- Convert critical PDFs to Markdown summaries (human-readable by agents)
- Fix catalog.py language detection for binary files
- Add text-based product specs, pricing sheets, and client profiles that agents can actually read

### 4.2 Session Amnesia

Every session restart = complete memory loss. Session #22 doesn't know what session #21 learned, discovered, or decided. This means:
- Repeated analysis work (this analysis has been partially done before)
- No learning from past mistakes
- No accumulated context about client conversations

**Fix:** Implement a session journal in `workspace/state/session-journal.md` that each session appends to with key decisions, findings, and action items. Next session reads it on startup.

### 4.3 Trello Integration is Unused

The Trello API helper exists (`tools/trello_api.sh`), the board is configured, but agents aren't actually calling it. Tasks are managed via filesystem files, and Trello goes stale. The Founder was told they can monitor on Trello, but it doesn't reflect reality.

**Fix:** Either integrate Trello updates into the PM workflow (create card on task receipt, move on status change) or drop Trello and use Telegram-only for monitoring. A tool the Founder checks that's always wrong is worse than no tool.

### 4.4 Model Routing Contradictions

- `VISION.md` says PM and Arch use Opus
- `CLAUDE.md` says PM should use Sonnet/Haiku for its own work
- `loop.sh` doesn't specify a model (defaults to whatever Claude Code uses)
- Agent CLAUDE.md files mention Opus 4.6 but startup scripts use Sonnet 4.5

This inconsistency means cost management is unpredictable. Nobody knows which model is actually being used.

**Fix:** Pick one source of truth for model selection. Update all files to match. Add model selection to the Claude command in loop.sh.

### 4.5 Shell Injection Vulnerability in Trello API

`trello_api.sh` uses Python string interpolation inside bash shell variables:
```bash
python3 -c "import urllib.parse; print(urllib.parse.quote('${card_title}'))"
```
If `card_title` contains a single quote, this breaks. If it contains malicious content, it could execute arbitrary Python code.

**Fix:** Pass variables as arguments, not string interpolation:
```bash
python3 -c "import urllib.parse, sys; print(urllib.parse.quote(sys.argv[1]))" "$card_title"
```

### 4.6 Telegram Bridge Edge Cases

- If a task starts with "Update the..." it gets classified as a reply (matches `REPLY_PATTERNS`: "update...to")
- Questions that are also tasks ("Can you create a pitch deck?") will be classified as tasks due to pattern priority, which is correct, but there's no feedback loop
- No support for file attachments (Founder can't send images, PDFs, or voice notes)
- Messages longer than 4000 chars get truncated — the Founder has no way to send long briefs

**Fix:** Add file attachment handling (save to `workspace/comms/from-founder/attachments/`). Consider splitting long messages into multi-part.

### 4.7 No Tests for Any Tools

Zero tests for:
- `telegram_bridge.py` (650 lines, complex classification logic)
- `trello_api.sh` (242 lines, API calls)
- `catalog.py` (244 lines, file processing)

Any change to these tools is flying blind.

**Fix:** Add basic test suites, at minimum for message classification logic and catalog generation.

## PART 5: STRATEGIC GAPS — THE BIG PICTURE

### 5.1 You're Building Too Many Things at Once

Active/planned products:
1. Bilal (Voice Agent) — in pilot
2. Baian (Chat Agent) — in pilot
3. HR Interview Agent — concept phase
4. Avatar Agent — prototype
5. Internal Platforms (CRM, PMS, HR, Ops) — pre-development
6. AI Workforce System — operational

That's 6 products/systems for a startup with **zero revenue and zero employees** (besides the Founder). This is classic startup dilution.

**Harsh truth:** Kill products 3, 4, and 5 from the active roadmap. They don't generate revenue. They distract from the 2 products that can (Bilal and Baian). Revisit them after you have 5+ paying customers.

### 5.2 No Go-to-Market Strategy

You have products, leads, and ROI data. What you don't have:
- Target customer profile (which companies, what size, what pain point)
- Sales pitch (30-second version, 5-minute version, 30-minute version)
- Pricing page or one-pager
- Demo booking system (Calendly)
- Follow-up automation
- Win/loss tracking

This isn't a marketing problem. It's a sales infrastructure problem. You need to be able to go from "here's a lead" to "here's a paying customer" in a repeatable process.

### 5.3 Single Point of Failure: The Founder

Everything flows through one person:
- Sales calls — Founder
- Product demos — Founder
- Technical decisions — Founder
- Partnership outreach — Founder
- Marketing approval — Founder

The AI Workforce can help with research, content, and analysis. But the actual revenue-generating activities (closing deals, running demos, building relationships) require the Founder's daily attention. If the Founder gets sick or busy, everything stops.

**Fix:** This isn't fixable right now, but it should inform priorities. The Founder's time is the scarcest resource. Allocate it to the highest-ROI activity: closing the 3 pilot clients into paid contracts. Everything else is secondary.
